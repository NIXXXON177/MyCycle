## Code freeze (1.7.5 RC)

**До результатов прогона на реальном устройстве — не менять код.**

Только документация и заполнение таблицы производительности по итогам тестов.

### Снимок RC (перед прогоном на устройстве)

Зафиксировать состояние кандидата **локально**, без push — чтобы при FAIL можно было чинить точечно, не теряя базовую точку RC.

> **Важно:** тег/ветка указывают только на **коммит**. Если RC ещё не закоммичен, снимок будет указывать на предыдущий релиз (1.7.0).

```bash
# 1. Ветка RC
git checkout -b release/1.7.5-rc1

# 2. Локальный коммит (без push — это не Release Decision)
git add .
git commit -m "Release Candidate 1.7.5: Reliability & Data Protection Update"

# 3. Тег RC
git tag 1.7.5-rc1

# 4. Собрать APK с этой точки для прогона на устройстве
flutter build apk --release
```

### Артефакт для валидации (заполнить после сборки)

Убедиться, что на устройстве тестируется APK **именно из RC-коммита**:

```
Коммит:    d1ad434 (release/1.7.5-rc1, tag 1.7.5-rc1)
Версия:    1.7.5+1 (pubspec.yaml)
Сборка:    YYYY-MM-DD HH:MM
Файл:      build/app/outputs/flutter-apk/app-release.apk
Размер:    ... байт (... МБ)
SHA-256:   ...
```

**Windows (SHA-256):**

```powershell
Get-FileHash build\app\outputs\flutter-apk\app-release.apk -Algorithm SHA256
```

**Linux / macOS:**

```bash
sha256sum build/app/outputs/flutter-apk/app-release.apk
```

> Сохранить хэш в заметках релиза или в отчёте с устройства — при retest можно сравнить, что установлен тот же артефакт.

Push и GitHub Release — **только после** зелёного отчёта (Release Decision).  
При **FAIL** → fix на ветке от `release/1.7.5-rc1` → retest → не переходить к v1.8.0, пока FAIL не закрыт.

## Перед push в `main`

1. Обновить версию в `pubspec.yaml` (`x.y.z+1`)
2. Заполнить `release_notes/whats_new.txt` (обязательно для CI)
3. Добавить запись в `CHANGELOG.md`
4. Запустить локально:
   ```bash
   flutter pub get
   flutter analyze lib
   flutter test --exclude-tags benchmark
   flutter test --tags benchmark   # опционально: stress-бенчмарк backup (~15 сек)
   ```
5. Пройти **ручной чеклист** на реальном устройстве (см. ниже)

## CI/CD

Push в `main` запускает workflow **Build and Release APK**:

- Сборка release APK
- Публикация GitHub Release с тегом `v<version>.<build_number>`
- Файлы: `MyCycle.apk`, `manifest.json` (для in-app updater)

## Версионирование

- **x.y.z** — версия в `pubspec.yaml` (семантика для пользователя)
- **build_number** — номер сборки GitHub Actions (`github.run_number`)

## Критерий выпуска 1.7.5

Релиз готов, когда на телефоне проходят **все** пункты:

- [ ] **Золотой сценарий** (см. ниже)
- [ ] **Обновление 1.7.0 → 1.7.5** поверх, без удаления
- [ ] **Backup → uninstall → restore** на том же устройстве
- [ ] **Restore на другом устройстве** (чистая установка)
- [ ] **Restore поверх существующих данных** (см. ниже)
- [ ] **Битый архив** — отказ без потери текущих данных
- [ ] **PIN после Restore** — тот же код разблокирует приложение
- [ ] **Нехватка места (ENOSPC)** — понятное сообщение при экспорте
- [ ] **Таблица производительности** — профили малый / средний / большой
- [ ] `flutter test` — backup, restore, migrations

### Золотой сценарий

1. Установить приложение
2. Заполнить цикл, самочувствие, 5–10 записей дневника с фото
3. Добавить хотелки и важную дату
4. Настроить PIN и биометрию
5. Экспорт Backup v2 (проверить превью: размер, фото, записи)
6. Удалить приложение → установить заново → Restore
7. Убедиться: состояние «после» неотличимо от «до удаления»

**Критерий успеха:** пользователь не замечает разницы.

### Обновление без удаления (OTA)

Большинство пользователей обновятся именно так:

1. Установить **1.7.0**, создать данные
2. Установить **1.7.5 поверх** (APK из GitHub Releases)
3. Проверить: все данные на месте
4. Сделать Backup v2 → Restore

Проверяет: миграции, совместимость старых данных, новый Backup v2.

### Restore поверх существующих данных

Типичный сценарий реального пользователя — не чистая установка, а **замена** уже имеющейся базы.

```
Устройство B
    ↓
Уже есть свои данные
    ↓
Restore ZIP с другого устройства (или старого бэкапа)
    ↓
Подтверждение замены в диалоге
    ↓
Данные из ZIP полностью заменяют текущие
```

**Покрытие в коде:** перед заменой `BackupService` сохраняет снимок в rollback-директорию (`_backupCurrentState` → `_rollbackFrom` при ошибке). Автотест `restore rolls back when archive database is invalid` проверяет откат.

**На устройстве проверить:**

1. Устройство B: создать **свои** данные (цикл, 2–3 записи дневника)
2. Импортировать валидный ZIP с устройства A
3. Подтвердить замену → данные A на месте, данных B нет
4. *(опционально)* Повторить с **битым** ZIP → текущие данные B **не повреждены** (откат)

- [ ] Успешный restore заменяет старые данные содержимым ZIP
- [ ] При ошибке restore откатывается к состоянию до импорта

### PIN и биометрия после Restore

**PIN:** восстанавливается из `settings.json` как `pinHash` (SHA-256 с солью приложения). В ZIP нет plaintext PIN. Старые архивы с `pinCode` по-прежнему поддерживаются — при импорте хэшируются.

**Биометрия:** после Restore **всегда выключается**. Ключи на Android/iOS привязаны к устройству; при переносе на другое устройство включённый флаг без ключа ведёт к непредсказуемому поведению. Пользователю показывается: «Биометрическую аутентификацию необходимо настроить заново.»

При Restore обязательно проверить:

- [ ] PIN разблокирует приложение (тот же код, что до бэкапа)
- [ ] Биометрия **выключена** после Restore; включается вручную в настройках
- [ ] После Restore `securityProvider` обновлён (перезапуск приложения)

### Размер резервной копии и производительность

Замеры на **реальном устройстве** (секундомер + размер `.zip` в файловом менеджере).  
Время: от нажатия «Экспортировать» / «Восстановить» до snackbar успеха.

#### Профили данных (заполнить после прогона на устройстве)

| Профиль | Фото | Дневник | Размер ZIP | Экспорт | Restore |
|---------|------|---------|------------|---------|---------|
| Малый | 20 | 50 | ? | ? | ? |
| Средний | 100 | 300 | ? | ? | ? |
| Большой | 300 | 1000 | ? | ? | ? |

**Как получить профили:**

| Профиль | Как набить данные |
|---------|-------------------|
| Малый | Вручную за 10–15 минут |
| Средний | Вручную или частично через dev-seeder |
| Большой | Dev mode: 7 тапов по «Версия» → stress-данные (300 фото / 1000 дневник / 500 самочувствие) |

**Устройство:** _модель, Android, версия MyCycle_  
**Дата замера:** _ГГГГ-ММ-ДД_

> Даже приблизительные значения полезны — через несколько релизов появится история производительности.

#### Справочно: desktop / CI (не сравнивать с телефоном)

Запуск: `flutter test --tags benchmark`  
FFI, placeholder JPEG ~200 Б — размер архива **не** отражает реальные фото.

| Платформа | Экспорт | Размер архива | Восстановление |
|-----------|---------|---------------|----------------|
| Windows (dev, 2026-06-19) | 0,3 сек | ~219 КБ | 2,9 сек |

#### Нехватка места на диске

При экспорте ZIP временно нужно **~2×** итогового размера (сборка архива в памяти + запись файла). Превью экспорта показывает оценку пикового потребления.

При нехватке места приложение показывает:

- экспорт: «Недостаточно свободного места для создания резервной копии.»
- восстановление: «Недостаточно свободного места для восстановления резервной копии.»

Проверить на устройстве (по возможности): экспорт при заполненной памяти → понятное сообщение, данные не повреждены.

## Frozen until v2.0

Следующие компоненты **заморожены** до v2.0. Любое изменение ломает совместимость или требует миграции у существующих пользователей:

| Компонент | Текущее состояние |
|-----------|-------------------|
| Backup format | v2 (`backupVersion: 2`) |
| Database schema | v5 |
| Migration chain | v1 → v5 |
| Security model | PIN hash + biometric reset on restore |

**Не менять без крайней необходимости и без плана миграции для v2.0.**

Дополнительно до v2.0 не добавлять:

- новые таблицы SQLite
- новые типы данных в backup
- новые настройки, влияющие на `settings.json`
- новые разрешения Android

Новые функции v1.8.x строятся **только** на чтении существующих таблиц.

## Release Decision

**До зелёного чеклиста на устройстве — не менять архитектуру и не откладывать релиз ради «ещё пары мелочей».**

Если:

- ✓ Золотой сценарий
- ✓ OTA 1.7.0 → 1.7.5
- ✓ Restore на другом устройстве
- ✓ Restore поверх существующих данных
- ✓ Битый архив (откат без потери данных)
- ✓ PIN после Restore
- ✓ ENOSPC
- ✓ Таблица производительности
- ✓ Нет блокирующих багов

Тогда:

→ **Release Decision**  
→ **Commit**  
→ **Push**  
→ **GitHub Release**  
→ **`feature/v1.8.0-year-review`**

**Тег / название релиза 1.7.5:**

```
v1.7.5
Reliability & Data Protection Update
```

(CI создаст тег `v1.7.5.<build_number>` автоматически при push в `main`.)

**Сразу после релиза** — ветка для v1.8.0, без смешивания с hotfix 1.7.5:

```bash
git checkout -b feature/v1.8.0-year-review
```

Не искать десяток мелких улучшений после успешного тестирования — это главная причина бесконечно откладываемых релизов.

## Отчёт с устройства (шаблон)

Скопировать, заполнить после прогона, прислать перед Release Decision.

### Интерпретация статусов

| Статус | Значение |
|--------|----------|
| **PASS** | Работает как задумано |
| **PASS WITH NOTES** | Работает, но есть нюансы UX или производительности (не блокирует релиз) |
| **FAIL** | Блокирует релиз — нужен fix и повторный прогон |

> Сбой на тесте — не провал проекта, а сигнал найти проблему **до** публикации.

### Шаблон

```

Дата: YYYY-MM-DD

APK (RC): d1ad434 / 1.7.5-rc1
SHA-256: ...

Золотой сценарий: PASS / PASS WITH NOTES / FAIL
OTA 1.7.0 → 1.7.5: PASS / PASS WITH NOTES / FAIL
Restore другое устройство: PASS / PASS WITH NOTES / FAIL
  Примечание: ...
Restore поверх данных: PASS / PASS WITH NOTES / FAIL
  Примечание: ...
Битый архив: PASS / PASS WITH NOTES / FAIL
PIN после Restore: PASS / PASS WITH NOTES / FAIL
ENOSPC: PASS / PASS WITH NOTES / FAIL / SKIP

Профили:

  ZIP: ...
```

**Пример PASS WITH NOTES:**

```
Restore другое устройство: PASS WITH NOTES
  Примечание: после восстановления требуется перезапуск приложения
  для обновления виджета.
```

### Release Decision

- Все **критичные** сценарии — PASS или PASS WITH NOTES (без FAIL) → **Release**
- Любой **FAIL** → **Fix and retest**, релиз не выпускать
- ENOSPC допускает **SKIP**, если проверить не удалось

После **Release** → Commit → Push → GitHub Release → `feature/v1.8.0-year-review`.
